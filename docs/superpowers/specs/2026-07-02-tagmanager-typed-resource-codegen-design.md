# Tag Manager typed-resource codegen (Phase 5)

## Background

The original foundation design (`2026-06-30-matomo-tagmanager-provider-design.md`, §6, §6.4, §12) planned two resource forms per Tag Manager entity kind: a generic fallback (`matomo_tagmanager_{tag,trigger,variable}`, type string + `parameter` blocks) and a set of typed resources, one per built-in Matomo type (e.g. `matomo_tagmanager_tag_customhtml`), generated by a `tools/gen` program. Only the generic fallback was built (Phase 4, PR #3/#4/#5). This spec covers Phase 5: building `tools/gen` and generating the typed resources.

This is an explicit correction of an earlier, now-deleted spec (`2026-07-02-tagmanager-type-coverage-design.md`) which mistakenly proposed adding acceptance-test coverage to the *generic* resources instead. The generic resources stay in place, untouched, as the fallback for types the generator doesn't recognize (e.g. third-party-plugin-contributed types) — this phase supplements them, it does not replace them.

## Goals

- Generate one typed Terraform resource per Matomo-built-in, creatable Tag Manager type (tag/trigger/variable), for the `web` context, covering every such type in one pass.
- Each typed resource exposes the type's actual parameters as named, typed Terraform attributes (not a generic `parameter` block), pulling schema (names, types, defaults, available values, descriptions) directly from Matomo's own discovery API rather than hand-transcribing it from source.
- Keep the codegen self-verifying: CI regenerates and diffs against committed output, so schema drift from a future Matomo version is caught automatically.

## Non-goals

- `android`/`ios` contexts (may be a follow-up phase; out of scope here).
- Read-only `PreConfigured` variables (~70 built-ins with no create path — never returned by the discovery API's variable endpoint, so nothing to generate).
- Removing or changing the generic `matomo_tagmanager_{tag,trigger,variable}` resources or their client code.
- Encoding every nuance of Matomo's `condition` grammar beyond `&&`/`||`/`!`/`==`/`!=` against literals (if a real type's condition string doesn't parse under this grammar, generation fails loudly rather than guessing — see "Condition parsing").

## Architecture

`tools/gen` is a Go program (`go run ./tools/gen`) that:

1. Connects to a live Matomo instance via `internal/matomo.Client`, using `MATOMO_BASE_URL`/`MATOMO_API_TOKEN` (same env vars and fixture the acceptance-test suite already uses).
2. Calls `TagManager.getAvailableTagTypesInContext`, `getAvailableTriggerTypesInContext`, `getAvailableVariableTypesInContext` for `idContext="web"`. Each call already returns full per-type parameter metadata inline (via `TemplateMetadata::formatTemplates()` → `BaseTemplate::toArray()` → `SettingsMetadata::formatSetting()` per parameter) — no further per-type fetch is needed.
3. Looks up each type's required-parameter list in a hand-maintained annotation table (`tools/gen/required.go`) — see "Required-field annotations".
4. Parses each parameter's `condition` string into a validator spec — see "Condition parsing".
5. Emits one schema file per type into `internal/provider/generated/`.
6. Emits `internal/provider/generated/generated_resources.go`, a single file with one `resource.Resource` constructor entry per generated type, referenced from `provider.go`'s `Resources()` list.
7. Emits one acceptance-test scaffold per type, but only if that file does not already exist (never overwrites a hand-edited test).

A single hand-written file per kind (`internal/provider/typed_tag_resource.go`, `typed_trigger_resource.go`, `typed_variable_resource.go`) implements the CRUD logic once (`Create`/`Read`/`Update`/`Delete`/`Metadata`/`Configure`/`ImportState`), reusing the same shared helpers the generic resources already use (`resolveDraftVersionID`, `buildEntityID`/`parseEntityID`, `bareEntityIDs`, `compositeEntityIDs`). Each generated schema file supplies a small model type implementing a `toParams()`/`fromParams()` interface that the shared CRUD logic calls into, instead of the generic resource's `parameter`-block conversion.

## Discovery & schema extraction

Per type, the discovery API returns:
- `id` — the type identifier Matomo's own `addContainerTag`/`addContainerTrigger`/`addContainerVariable` expect as `type` (e.g. `CustomHtml`). Used as the authoritative slug source for the generated resource name (`matomo_tagmanager_tag_customhtml`) and Go identifiers, since it's guaranteed unique per kind (unlike `name`, which is only used for the schema's human-readable `Description`).
- `parameters[]`, each with `name` (→ `snake_case` Terraform attribute name, e.g. `customHtml`→`custom_html`), `type` (`string`/`bool`/`array`/etc. → Terraform attribute type, mapped by this semantic `type` field, not by the presentation-only `uiControl`), `defaultValue`, `availableValues` (→ `OneOf` validator when non-empty), `description` (→ schema `Description`), `condition`.

Variable templates are pre-filtered by Matomo itself to exclude `isPreConfigured()` types, so the ~70 read-only built-ins never appear in the generator's input.

## Required-field annotations

Verified (by reading `SettingsMetadata::formatSetting()`'s full body directly from Matomo source) that required-ness is **not exposed anywhere** in the discovery API response: `FieldConfig::$validators` (where a type's `NotEmpty()` etc. validators are actually attached, server-side) is never serialized into the API's returned array, in Tag Manager's own template metadata or the generic Settings API it's built on. There is no way to derive required-ness from data Matomo exposes.

`tools/gen/required.go` is therefore a hand-written Go source file (not generated), one map entry per type, populated by reading that type's Matomo PHP source once (the same process used for `Constant`/`CustomHtml` earlier this project):

```go
var requiredParams = map[string]map[string][]string{
	"tag": {
		"CustomHtml": {"customHtml"},
	},
	"variable": {
		"Constant": {"value"},
	},
}
```

`tools/gen` hard-fails (`log.Fatalf`) if it discovers a type with no entry in this table — a type can only ship as "no required params" if someone explicitly wrote an empty slice for it, never by omission. Required parameters get `Required: true` in the generated schema; others get `Optional: true` (plus `Computed: true` where Matomo supplies a real, non-empty default).

## Condition parsing

`condition` strings use a small grammar over field names: identifiers, `!`, `&&`, `||`, `==`/`!=` against literal values (e.g. `sitesearch && !use_sitesearch_default`, `triggerType == "pageview"`) — confirmed against `FieldConfig::$condition`'s docblock in Matomo source. `tools/gen/condition.go` parses this into a small AST (`Ref`, `Not`, `And`, `Or`, `Eq`), then translates it into `terraform-plugin-framework` validators attached to the dependent attribute:

- `Ref(x)` → `AlsoRequires(path.MatchRoot("x"))`.
- `Not(Ref(x))` → `ConflictsWith(path.MatchRoot("x"))`.
- `Eq(Ref(x), literal)` → `conditionEqualsValidator` (hand-written once, shared across all generated files) — checks the referenced attribute's configured value equals the literal.
- `And` → attach both resulting validators. `Or` → `conditionAnyOfValidator` (hand-written once), since terraform-plugin-framework has no native "satisfy any of" validator.

If a real condition string doesn't parse under this grammar, `tools/gen` fails loudly rather than silently dropping the constraint. The parser gets unit tests against real condition strings pulled from live Matomo templates, not synthetic examples.

## File layout and naming

- Resource type names: `matomo_tagmanager_{tag,trigger,variable}_<lowercased id>`, e.g. `matomo_tagmanager_tag_customhtml`, `matomo_tagmanager_trigger_pageview`, `matomo_tagmanager_variable_constant`.
- Generated schema files: `internal/provider/generated/tag_customhtml.go`, one per type, containing:
  - A model struct with `tfsdk` tags (common fields `id`/`container_id`/`name`/`status`/`fire_trigger_ids`/`block_trigger_ids` for tags, analogous common fields for triggers/variables, plus type-specific fields below).
  - A `schema.Schema` value.
  - `toParams()`/`fromParams()` methods converting between the model and the `map[string]string` the existing `matomo` client package already expects.
- `internal/provider/generated/generated_resources.go`: one constructor entry per type, wired into `provider.go`'s `Resources()`.
- Generated acceptance-test scaffolds: `internal/provider/generated/tag_customhtml_acc_test.go` — created only if absent; a minimal create+read-back test using placeholder values for required attributes (strings: `"test-value"`, bools: `true`, selects: first `availableValues` entry). Never regenerated once present, so hand-improved tests (e.g. today's real `CustomHtml` test with actual HTML content) survive future `tools/gen` runs untouched.

## Shared runtime

`internal/provider/typed_tag_resource.go` (and `_trigger_`/`_variable_` counterparts) are hand-written once. Each defines a non-generic `typedTagResource` struct holding a `meta typedTagMeta` (type ID, resource name, schema) and a model-constructor function, instantiated once per generated type in `generated_resources.go`. `Create`/`Read`/`Update`/`Delete` bodies mirror today's generic `matomo_tagmanager_tag` resource logic (draft-version resolution, composite IDs, fire/block trigger handling, not-found detection) but call the model's `toParams()`/`fromParams()` instead of the generic resource's `parameter`-block conversion, and pass `meta.typeID` as the fixed `Type` rather than reading it from user config.

## CI wiring

`.github/workflows/acceptance.yml` gets a new step, before the existing acceptance-test run, that executes `go run ./tools/gen` against the same Matomo fixture, then `git diff --exit-code -- internal/provider/generated/`. A non-empty diff fails the build with a message pointing at re-running the generator and committing the result. This runs on every push/PR, so schema drift introduced by a Matomo upgrade (or a source edit without regenerating) is caught immediately rather than silently shipping stale typed resources.

## Testing

Every generated type gets at least the generated placeholder-value create+read-back acceptance test described above, following the existing "one site+container per test" pattern (no shared fixtures across tests). Types needing more meaningful coverage (real content, condition interactions, etc.) get their scaffold hand-edited once; `tools/gen` will not overwrite an existing test file on subsequent runs.
